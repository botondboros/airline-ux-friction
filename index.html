Airline Booking UX Friction Analyzer
This study measures the friction encountered when purchasing a one-way ticket online across sixteen European airlines on the BUD → London route. Each flow is scored as a Friction Index starting at zero — friction events add points, positive UX signals reduce them. A lower index means a smoother, more transparent purchase experience.

Friction Index — How it works

The Friction Index measures the cumulative difficulty of completing an online ticket purchase, from the homepage through to the payment confirmation screen. Unlike a satisfaction score, it is additive: every friction event encountered increases the index, while positive UX signals reduce it. A score of zero would represent a perfectly frictionless flow. There is no theoretical upper bound.

The index is designed to be comparable across airlines on the same route and fare type. All measurements are taken on a BUD → London booking (one-way, 1 adult, ~3 weeks from run date, cheapest available fare class) using an automated Playwright browser session running at 1440×900 viewport.

Calculation formula

Friction Index = 0 (start)
For each friction event: FI += event weight
For each positive signal: FI = max(0, FI − bonus weight)
Final FI = sum of all deductions minus sum of all bonuses (floor 0)

Event weights

Weights were set based on severity of user impact. Dark patterns (pre-ticked items, confirmshaming, hidden declines) carry the highest friction weights because they involve deliberate UI manipulation. Price changes and late fees also carry high weights because they break the user's mental model of cost formed at the search stage.

ACCOUNT_PUSH +4 flow A loyalty programme or account registration banner interrupts the booking flow outside of the passenger details form.
LOYALTY_PUSH +3 flow A loyalty programme is promoted on the passenger details form or results page, adding cognitive load to an already-dense step.
SLOW_PAGE +3 technical A page load exceeding 4 seconds during the booking flow, measured from navigation trigger to interactive state.

FORM_FIELD_EXTRA +1 flow Each required passenger form field beyond a baseline of 5 fields (first name, last name, email, phone, date of birth).
GUEST_CHECKOUT -2 positive A guest checkout path is clearly and prominently available, not requiring account creation to complete a purchase.
CLEAR_SKIP -2 positive A skip or "no thanks" option for an optional step (seat selection, bag, upsell) is clearly visible and easy to activate.
PRICE_TRANSPARENT -2 positive The final price displayed at checkout exactly matches what was shown at the search/results stage with no late additions.

Data collection

Each booking flow is instrumented with a Playwright script that logs every screen, click, form field, popup, and price shown. Screenshots are taken at each step. The StepLogger class records friction events in real time as the scraper navigates the flow, producing a structured JSON log per airline.

A Claude Vision analysis layer then reviews each screenshot and independently scores UI complexity, price visibility, and CTA clarity. Dark pattern detection runs both programmatically (DOM inspection for pre-ticked checkboxes, hidden elements, price mutations) and visually (Claude identifies confirmshaming copy, buried fare options, and ambiguous skip affordances).

Limitations: airline websites change frequently and selectors may require updating between runs. Automated scrapers may encounter anti-bot measures on some carriers. Demo data shown here represents researched estimates of typical flow behaviour; run npm run full-run to collect live measurements.
